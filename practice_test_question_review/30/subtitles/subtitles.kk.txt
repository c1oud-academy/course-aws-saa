Қайырлы күн, құрметті студенттер! Біз Practice Test сұрағын талдауды бастаймыз. Не берілген? Компания кейбір құпия салық деректерін AWS жүйесінде сақтауды жоспарлап отыр. Бұл деректер жалпыға қолжетімді болмауы керек. Сонымен қатар, бұл деректерге қол жеткізу тек белгілі бір VPC үшін қол жетімді болуы керек. Екінші талап. Бұл сақтау орнындағы объектілер жойылудан және қайта жазылудан қорғалуы керек. Осылайша, WORM, яғни write-once-read-many деген реттеуші талабы сақталуы керек. Оның идеясы, бір рет жазылады, көп рет оқылады. Бұл объектілерді қайта жазуға болмайды. Ұсынылған жауап нұсқаларынан осы талапты жүзеге асыруға көмектесетін екі жауап нұсқасын таңдауымыз керек. Бізге қандай жауап нұсқалары берілгенін көрейік. Бірінші және төртінші жауап нұсқаларының дұрыс екенін көреміз. Сонымен қатар, нұсқалардың ешқайсысы бір-біріне ұқсамайтынын көреміз. Сондықтан бұл сұрақ күрделі болып саналады және AWS жайлы теорияны жақсы білуді талап етеді. Жауап нұсқаларының әрқайсысын талдап, оның неге дұрыс немесе жалған екеніне жауап берейік. Теорияны, атап айтқанда Amazon S3 сервисінің Access Points функционалдығын еске түсірейік. Бұл - объектіні оқу немесе жазу сияқты операциялар үшін S3 Bucket-ке кіруді шектеуге мүмкіндік беретін функция. Суретте сіз сол жақ шарлау мәзірінде Amazon S3 сервисінің Access Points таңдалғанын және кейбір кіріс деректері, атап айтқанда Bucket көрсетілгенін көре аласыз. Төменде шектеулер қойылатын VPC тізімі берілген. Барлық басқа көрсетілмеген VPC үшін бұл S3 қол жетімді болмайды. Тағы бір мәселе, Access Points артықшылығы олар Интернетке қосылуды қажет етпейді. Барлық деректер алмасу жаһандық AWS инфрақұрылымында орын алады. Мұның бәрі қосымша мүмкіндіктер береді, атап айтқанда, Access Points арқылы S3 Bucket-ке аймақаралық қатынасты орнатуға болады. Бұл да мүмкін. Бұл деректеріңізді жылдамырақ тасымалдауға көмектесу үшін AWS Global Accelerator пайдаланады. Белгілі бір тапсырманы орындау үшін маңызды тағы бір функционал - S3 Object Lock. Идея мынада: бұл функция S3 Bucket ішіндегі объектілерді жазуға және қайта жазуға тыйым салады және бұл объектілерді шексіз рет оқуға мүмкіндік береді. Белгілі бір нюанстар бар. Object Lock S3 bucket жасалған кезде бапталынады. Жасау кезінде Advanced settings қойындысында Object Lock опциясында Enable таңдау керек. Object Lock функциясын қосқаннан кейін оны кейін өшіру мүмкін емес екенін ескеріңіз. Сонымен қатар, ол автоматты түрде Bucket versioning қосады. Бізде Object Lock қосулы кезде, Bucket versioning өшіру немесе тоқтату мүмкін емес. Сондықтан дұрыс жауаптар – белгілі бір VPC-ге қол жеткізуді шектеу үшін Amazon S3 Access Points пайдалану. Екінші жауап – файлдарды бір рет жазып, қажет мөлшерде оқи алу нормативтік талаптарды ескеріп S3 bucket жасау кезінде Object Lock функциясын пайдалану. Енді дұрыс емес жауап нұсқаларын толығырақ қарастырайық. Бірінші жағдайда біз AWS Network Firewall туралы айтамыз. Network Firewall S3 Bucket деңгейінде емес, VPC деңгейінде бапталынады. Осылайша, VPC үшін деректер ағынын баптай алсақ та, біз Network Firewall арқылы S3 bucket-іне кіруді шектей алмаймыз. Сондықтан, бұл жауап нұсқасы дұрыс емес. Егер біз S3 Glacier туралы айтатын болсақ, ол да сәйкес келмейді, өйткені Glacier мұрағатталған деректерді сақтау және деректерді бірден емес, біраз уақыттан кейін алу үшін қолданылады. Ал бізге қажет болғанда, бұл деректерді сол мезетте алу мүмкіндігіміз болуы керек деп айтады. Тағы бір мәселе, Glacier жаңа файлдарды қайта жазуға немесе жасауға тыйым салмайды. Осылайша, WORM талабы, яғни write-once-read-many деген талап орындалмайды. Егер біз соңғы Object versioning өшіріп, Object Lock опциясын қосу нұсқасы туралы айтатын болсақ, ол өзін-өзі жоққа шығарады, себебі Object Lock қосқан кезде Object versioning автоматты түрде қосылады. Object Lock опциясын қосқаннан кейін оны өшіруге немесе тіпті тоқтатуға болмайды. Сондықтан, бұл нұсқа да дұрыс емес. Осымен Practice Test сұрағын қарастырдық. Назар аударғаныңызға рахмет.
