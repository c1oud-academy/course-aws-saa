 Добрый день, уважаемые студенты! Мы с вами начинаем разбор вопросов практик теста. Согласно вопросу, у компании есть различные департаменты в различных частях мира. Компания хочет получить IT-контроль над всей инфраструктурой и также хочет видеть все расходы. Если говорить со стороны департаментов, которые находятся в различных частях мира, они хотят оставить у себя административный контроль над своими ресурсами и также требуют, чтобы их ресурсы не пересекались с ресурсами других департаментов. Что необходимо сделать для того, чтобы оставить автономность департаментов, но предоставить возможность центрального IT-управления и видения всех расходов в одном месте. Нам необходимо выбрать два правильных ответа из предложенных. На этом слайде вы видите варианты ответов, также видите, что второй и пятый являются верными. Тем не менее, давайте остановимся на каждом из вариантов ответов и подробно разберем. Если говорить про первый вариант ответа, здесь предлагается создать отдельные availability-зоны для каждого из департаментов, а также использовать AWS Global Accelerator, чтобы улучшить связь между двумя availability-зонами. Этот вариант ответа является абсолютно не логичным. Почему? Потому что в первом предлагается создать availability-зоны, на самом деле регионы и availability-зоны внутри регионов создают AWS, у нас нет на это никакого влияния. Другой момент, количество availability-зон ограничено и бесконечно добавлять ваши департаменты в один регион не получится. Другой момент здесь говорится использовать AWS Global Accelerator, чтобы улучшить связь между двумя availability-зонами. Мы с вами помним, что AWS Global Accelerator используется для ускорения загрузки данных в S3 в ближайший регион. Далее попав в ближайший регион, она через внутреннюю глобальную инфраструктуру AWS быстрее доходит до конечного AWS региона. Это помогает ускорить загрузку. AWS Global Accelerator относится к cross-region действиям, а не к cross-availability zones. Поэтому этот вариант полностью мы исключаем. Если говорить про второй вариант, здесь предлагается использовать AWS Consolidated Billing в сервисе AWS Organizations, что позволит все AWS аккаунты различных департаментов объединить под одним основным корпоративным AWS аккаунтом. В этом AWS аккаунте будут видны какие аккаунты есть, какие расходы на каждом из этих аккаунтов, но при этом эти аккаунты могут отдельно использоваться владельцами этих аккаунтов ниже. Похоже, что этот вариант подходит. Давайте посмотрим другие варианты. В третьем варианте предлагается использовать AWS Trusted Advisor и AWS Resource Groups Tag Editor. AWS Trusted Advisor является сервисом, который помогает проверить, действительно ли наша инфраструктура соответствует best practices, там где это можно автоматически проверить. И Resource Groups Tag Editor – это тот сервис, который позволяет группами тегировать ваши ресурсы. Оно никак не относится к тому, чтобы мы отделили ресурсы различных департаментов друг от друга. Поэтому этот вариант не совсем подходит. Давайте посмотрим четвертый вариант ответа. Здесь предлагается настроить несколько разных VPC для каждого департамента в рамках одного общего AWS аккаунта. Далее предлагается запустить Transit Gateway с некоторой настройкой и настроить VPN-туннели между VPC. Здесь также этот вариант логически не совсем уверен. Здесь говорится, что мы должны использовать VPN-туннели между VPC. На самом деле для подключения VPC между собой мы используем VPC Peering. VPN отдельно настраивать для этого не нужно. Другой момент – это то, что все департаменты останутся жить в одном AWS аккаунте. Таким образом, вы расходы увидите общие в конце месяца. Здесь нужно было в рамках вопроса предоставить автономность каждой из департаментов. Давайте посмотрим теперь на пятый вариант ответа. Здесь предлагается настроить Cross Account доступ для администраторов, чтобы можно было управлять Child Accounts. То есть аккаунты, которые предназначены отдельно для каждого департамента. Исходя из этого разбора мы видим, что второй и пятый варианты ответа являются наиболее подходящими. Более того, они дополняют друг друга. То есть первое объединяет все аккаунты под одним общим аккаунтом и создает видимость всех расходов и видимость всех существующих используемых AWS аккаунтов. А пятый дает нам возможность центрально управлять и получать доступ к любому из AWS аккаунтов, департаментов, раскиданных по всему миру.
